<GDD_UPDATE_PROTOCOL>
## Протокол обновления GDD

### Шаг 1 — Подготовка
1. Прочесть текущую версию `docs/GDD/GDD_[Name]_v[N].md`
2. Проверить `brain/gdd_summary.md` на актуальность
3. Получить изменения от команды (DEV/ART/UIUX отчёты)

### Шаг 2 — Внесение изменений
1. Обновить соответствующие блоки GDD
2. Increment версии: v[N] → v[N+1]
3. Обновить changelog в начале документа

### Шаг 3 — Интеграция
1. Обновить `brain/gdd_summary.md` (краткая версия)
2. Создать артефакты для команды:
   - `GD_to_DEV_[DATE].md` — изменения для DEV
   - `GD_to_ART_[DATE].md` — изменения для ART
   - `GD_to_UIUX_[DATE].md` — изменения для UIUX
3. Записать в `decisions_log.md`

### Шаг 4 — Уведомление
1. Обновить `task_board.md` (если есть новые задачи)
2. Уведомить PM через `/sync` или напрямую
</GDD_UPDATE_PROTOCOL>

<FEATURE_SPEC_PROTOCOL>
## Протокол написания Feature Spec

### Триггер
PM ставит задачу TASK-GD-XXX: «Спроектировать [фичу]»

### Pre-Analysis (5 вопросов)
Перед написанием спека ответить письменно:

1. **Player Promise:** как эта фича усиливает обещание игры?
2. **Core Loop fit:** куда встраивается (micro/meta/macro)?
3. **KPI hypothesis:** какую метрику целим и на сколько %?
4. **Unity cost:** сложность Low/Med/High/Epic? perf-риски?
5. **Design heuristics:** какие правила поддерживаются/нарушаются?

### Feature Spec Структура
Создать `docs/GDD/FeatureSpec_[FeatureName]_v0.1.md`:

```markdown
# Feature Spec: [НАЗВАНИЕ]
# [STATUS:DRAFT] | Версия: 0.1 | Дата: [ДАТА]

## 1. Overview
- Feature Name: [название]
- Type: New Feature / Improvement / LiveOps / Economy / UX
- Player Promise fit: [одно предложение]
- Loop layer: Micro / Meta / Macro / LiveOps

## 2. Problem / Opportunity
[2-3 предложения]

## 3. Solution
### Описание
### User Flow
### Edge Cases

## 4. Progression & Economy Impact
### Новые ресурсы
### Balance constants
### Fairness check

## 5. KPI & Metrics
- Primary metric: [название]
- Baseline: [HYPOTHESIS или данные]
- Expected impact: [+/- N%]
- Success threshold: [значение]
- Rollback trigger: [значение]

## 6. Design Heuristics Check
- ✅/⚠️/❌ Player Promise
- ✅/⚠️/❌ Clarity
- ✅/⚠️/❌ Agency
- ✅/⚠️/❌ Fair Challenge

## 7. For Agents
### → DEV
- Systems: [список]
- ScriptableObjects: [список]
- Analytics events: [event_name | params]
- Performance estimate: [Low/Med/High] | ~[N] дней

### → UIUX
- Screens: [список]
- UX Flow: [граф]
- New UI elements: [список]

### → ART
- New assets: [список]
- Animation needs: [список]
- Placeholder OK: [да/нет]

### → PM
- Dependencies: [TASK-IDs]
- Recommended sprint: [Sprint N]
- Risk: [Low/Med/High]

## 8. Open Questions
- [ ] [Вопрос 1]
- [ ] [Вопрос 2]
```

### Integration
1. Если фича принята → добавить блоки в GDD
2. Запустить `/gdd-update`
3. Создать A2A артефакты
4. Вызвать `/done [TASK-GD-ID]`
</FEATURE_SPEC_PROTOCOL>

<BALANCE_REVIEW_PROTOCOL>
## Протокол Balance Review

### Когда вызывать
- После каждого milestone
- При изменении монетизации
- При подготовке к soft-launch
- При просадке ARPDAU или retention

### Анализ по блокам

#### A. Currency Flow
Для каждой валюты:
- Sources (откуда приходит)
- Sinks (куда уходит)
- Balance: surplus / balanced / deficit
- Risk: инфляция / дефицит / нет проблем

#### B. Progression Curve
- Формула прогрессии
- Время на уровень
- Мёртвые зоны
- First paywall timing

#### C. Monetization Health
Для каждого IAP оффера:
- Price и value proposition
- First appearance
- Fairness check (P2W риск)
- Pity/safety system

#### D. Ads Check
- Rewarded: добровольные? reward ценен?
- Interstitial: частота ≤ 1 per 3 min?

### Итоговый отчёт
Создать `docs/GDD/BalanceReview_[Date].md`:

```markdown
# Balance Review — [ДАТА]

## Executive Summary
[2-3 предложения]

## Currency Flow
[таблица]
Риски: [список]

## Progression Curve
[анализ]
Риски: [список]

## Monetization
[анализ]
Риски: [список]

## Recommendations
- P0: [критично]
- P1: [важно]
- P2: [желательно]
```

### GDD Update
Если найдены P0/P1 проблемы → обновить GDD
</BALANCE_REVIEW_PROTOCOL>

<DESIGN_REVIEW_PROTOCOL>
## Протокол Design Review

### Триггер
DEV или UIUX сообщают о завершении задачи

### Чек-лист ревью

#### Gameplay Review (DEV)
- [ ] Core mechanic реализована как в GDD?
- [ ] Win/Lose условия корректны?
- [ ] Feedback читаем (juice, audio, visual)?
- [ ] FTUE шаги соответствуют GDD?
- [ ] Нет scope creep?
- [ ] Analytics events реализованы?

#### Economy Review
- [ ] Balance constants соответствуют GDD?
- [ ] ScriptableObjects заполнены верно?
- [ ] Нет магических чисел в коде?

#### UX Review (UIUX)
- [ ] Screen flow соответствует GDD?
- [ ] HUD элементы присутствуют?
- [ ] Edge cases обработаны?
- [ ] Шрифты читаемы (min 14sp)?

### Решение

**PASS:**
```
## DESIGN REVIEW — PASS
Задача: [TASK-ID]
Результат: Реализация соответствует GDD.
Замечания: [список minor]
Решение: PM может закрывать задачу.
```

**PASS WITH NOTES:**
```
## DESIGN REVIEW — PASS WITH NOTES
Notes: [список TODO для следующего спринта]
Решение: Задачу можно закрыть.
```

**FAIL:**
```
## DESIGN REVIEW — FAIL
Блокер: [что не соответствует GDD]
Требуется: [исправление]
Приоритет: P0 / P1
Решение: Задача возвращается в IN_PROGRESS.
```

### После ревью
1. Сохранить в `.agent/artifacts/GD_review_[TASK-ID].md`
2. Записать в `decisions_log.md`
3. Уведомить PM
</DESIGN_REVIEW_PROTOCOL>

### Типы событий
- `progression` — доп. прогрессия (бонусные уровни, xp буст)
