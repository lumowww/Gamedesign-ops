---

## 🎯 ЦЕЛЬ СКИЛЛА

Превратить любой набор входящих материалов в **структурированный
Extraction Report** — манифест того, что берётся из каждого документа,
куда в текущий целевой документ это вставляется, и что остаётся
невыясненным после анализа.

**Ключевой принцип:** не копируй — переосмысляй.
Извлечённые данные адаптируются под текущую платформу,
механики и продуктовые цели проекта.

---

## 📥 ПОДДЕРЖИВАЕМЫЕ ФОРМАТЫ ВХОДНЫХ ДАННЫХ

| Формат | Что извлекаем |
|--------|--------------|
| `.md` / `.txt` | Весь текст, структура, списки, таблицы, ID-теги |
| `.pdf` | Текст (если не скан), таблицы, структура глав |
| `.jpg` / `.png` (схемы) | Диаграммы loop'ов, flow-чарты, архитектуры систем |
| `.jpg` / `.png` (таблицы) | Балансные таблицы, прогрессионные кривые, economy sheets |
| `.jpg` / `.png` (скриншоты UI) | Screen flow, HUD-элементы, UX-паттерны |
| `.jpg` / `.png` (питч-слайды) | Концепты, USP, аудитория, референсы |
| URL → GitHub / GDrive / Notion | Текстовый контент страницы, структура документа |
| Таблицы (CSV, XLS как изображение) | Числовые константы, balance curves, LTV данные |

### ═══ ФАЗА 3: МАППИНГ НА ЦЕЛЕВОЙ ДОКУМЕНТ ═══

**Цель:** для каждого блока данных определить точное место
в целевом документе.

**Целевые документы:**

- `GDD`          → маппинг на [ID:...] блоки
- `Feature Spec` → маппинг на секции 1–8
- `Balance Sheet`→ маппинг на [ID:BALANCE], [ID:ECONOMY]
- `LiveOps Event`→ маппинг на [ID:LO_...]
- `One-Pager`    → маппинг на блоки 1–11

#### EXTRACTION MAP

```markdown
## EXTRACTION MAP

*Целевой документ: [тип] | Текущая версия: [vN]*

| \# | Источник | Данные (кратко) | → Целевой [ID:] | Действие | Качество |
| :-- | :-- | :-- | :-- | :-- | :-- |
| 1 | doc.pdf §2 | «Core loop: tap→kill→loot» | [ID:CORE_LOOP] | ВСТАВИТЬ | [FACT] |
| 2 | scheme.jpg | Loop диаграмма | [ID:LOOPS_OVERVIEW] | ВОССОЗДАТЬ | [FACT]🟢 |
| 3 | brief.md | D7 target = 35% | [ID:LIVEOPS_KPI] | ВСТАВИТЬ | [FACT] |
```

**Легенда действий:**

- `ВСТАВИТЬ`    — данные подходят напрямую
- `АДАПТИРОВАТЬ`— суть берём, детали пересматриваем
- `ПРОВЕРИТЬ`   — требуют верификации перед вставкой
- `ВОССОЗДАТЬ`  — из изображения создаём текстовый блок
- `ИГНОРИРОВАТЬ`— нерелевантны или противоречивы
- `УТОЧНИТЬ`    — нужен вопрос к пользователю

## 🛠️ ИНСТРУКЦИИ АГЕНТУ

### Когда активировать

- Пользователь приложил ≥ 1 файл вместе с запросом
- Упомянул «старый GDD», «вот документы», «используй это»
- В запросе есть URL-ссылки
- Есть изображения в чате

### Порядок работы

1. Определи ЦЕЛЕВОЙ ДОКУМЕНТ (что пишем?)
→ Спроси если неочевидно
2. Фаза 1: зарегистрируй все входящие файлы
3. Фаза 2: проанализируй каждый файл своим анализатором
4. Фаза 3: построй EXTRACTION MAP
5. Фаза 4: Gap Analysis + Extraction Report
6. ТОЛЬКО ПОТОМ начни заполнять целевой документ
7. Выведи финальный Extraction Report

### Работа с изображениями (Claude multimodal)

- Описывай ВСЕ видимые элементы (текст, иконки, стрелки, числа, цвета)
- Сначала описывай → потом маппируй
- Нечёткий текст: `[НЕЧЁТКО: предположительно "..."]`
- Таблицы воссоздавай в Markdown даже если частично видны

### Приоритеты источников при конфликтах

Priority 1: Явные инструкции пользователя в чате
Priority 2: Файлы текущего запроса
Priority 3: brain/project_context.md
Priority 4: URL / внешние источники
Priority 5: Старые GDD / устаревшие документы

### Маркировка качества данных

- [FACT]       — из документа, без интерпретации
- [INTERPRET]  — интерпретировано из изображения/схемы
- [HYPOTHESIS] — предположение GD, не из документа
- [OUTDATED?]  — документ > 12 мес или признаки устарелости
- [CONFLICT:X] — противоречие с источником X
