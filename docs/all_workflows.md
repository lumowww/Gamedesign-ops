# WORKFLOW: NEW-PROJECT
# Файл: .agent/workflows/new-project.md
# Триггер: /new-project
# Агент: PM (инициирует), остальные подключаются по очереди

## PROTOCOL: ИНИЦИАЛИЗАЦИЯ НОВОГО ПРОЕКТА

Этот workflow запускается PM-агентом при старте нового проекта.

### Phase 1: Контекст (PM)
1. Запросить у пользователя:
   - Название проекта
   - Концепт (одно предложение или ссылка на референс)
   - Целевая платформа (iOS / Android / Both)
   - Дедлайн прототипа
   - Особые ограничения (нет в-app purchase / конкретный жанр / и т.д.)

2. Создать `project_context.md`:
```markdown
# PROJECT: [Name]
**Дата старта:** [дата]
**Дедлайн прототипа:** [дата]
**Концепт:** [одно предложение]
**Платформа:** iOS / Android
**Текущий Sprint:** Sprint 1
**Статус:** 🟡 ИНИЦИАЛИЗАЦИЯ
```

3. Создать первоначальный `task_board.md` с задачами для каждого агента

### Phase 2: GDD (GD)
GD получает задачу TASK-001: Создать GDD v0.1
- Минимальный GDD: Concept, Core Mechanic, Core Loop, Prototype Scope
- ETA: День 1

### Phase 3: Арт-библия (ART)
ART получает задачу TASK-002: Создать Арт-библию v0.1
- На основе GDD v0.1
- Цветовая палитра, mood board, стиль
- ETA: День 1-2

### Phase 4: UI Structure (UIUX)
UIUX получает задачу TASK-003: Создать список экранов
- На основе GDD: какие экраны нужны
- Создать Figma-борду структуру
- ETA: День 2

### Phase 5: DEV Setup
DEV получает задачу TASK-004: Настройка Unity проекта
- Unity 6 LTS, URP, базовые пакеты
- Структура папок (из 00_global.md)
- Подключить MCP-Unity сервер
- ETA: День 1

### Phase 6: PM — Синхронизация
После выполнения Phase 2-5:
- Провести /sync
- Обновить task_board.md
- Поставить задачи Sprint 1

# WORKFLOW: TASK-DONE
# Файл: .agent/workflows/task-done.md
# Триггер: /done [TASK-ID]
# Агент: любой

## ПРОТОКОЛ ЗАВЕРШЕНИЯ ЗАДАЧИ

При вызове `/done TASK-042`:

1. Обновить `task_board.md`: статус → DONE
2. Обновить соответствующий brain-файл:
   - GD → `gdd_summary.md`
   - ART → `art_bible_summary.md` или `asset_registry.md`
   - UIUX → `asset_registry.md` (UI часть)
   - DEV → `decisions_log.md`
3. Записать в `decisions_log.md`:
```
## [Дата] — TASK-[ID] завершена
**Агент:** [роль]
**Результат:** [что создано/изменено]
**Файлы:** [список изменённых файлов]
**Следующий шаг:** [TASK-ID] или ожидание PM
```
4. Уведомить PM: "TASK-[ID] завершена. Результат: [краткое описание]. Ожидаю новую задачу."

# WORKFLOW: ASSET-REQUEST
# Файл: .agent/workflows/asset-request.md
# Триггер: /asset-request [тип] [описание]

## ПРОТОКОЛ ЗАПРОСА АССЕТА

Используется DEV или UIUX когда нужен ассет которого ещё нет.

1. DEV/UIUX создаёт запрос:
```
## ASSET REQUEST — [Дата]
**Запросил:** DEV / UIUX
**Тип:** Sprite / Audio / Font / Particle / Prefab
**Описание:** [точное описание нужного ассета]
**Используется в:** [сцена / экран / система]
**Нужен к:** [дата / TASK-ID]
**Критичность:** P0 / P1 / P2
```

2. ART получает запрос → ищет бесплатный ассет или помечает для создания
3. ART обновляет `asset_registry.md` и сообщает статус

# WORKFLOW: GDD-UPDATE
# Файл: .agent/workflows/gdd-update.md
# Триггер: /gdd-update

## ПРОТОКОЛ ОБНОВЛЕНИЯ GDD

При обновлении GDD агентом GD:

1. Обновить `docs/GDD/GDD_[ProjectName]_v[N].md` (инкрементировать версию)
2. Записать изменения в diff-формате в `decisions_log.md`:
```
## GDD UPDATE v[N] — [Дата]
**Изменено:** [что именно]
**Причина:** [почему]
**Влияние на DEV:** [нужны ли изменения в коде]
**Влияние на ART:** [нужны ли новые ассеты]
**Влияние на UIUX:** [нужны ли новые экраны]
```
3. Обновить `gdd_summary.md` (краткое резюме макс. 500 слов)
4. Уведомить PM — PM перераспределяет задачи при необходимости
